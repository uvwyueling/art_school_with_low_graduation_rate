**表名：** `to_verify_17students.csv`

**说明：** 该表格由 `./scripts/special_student_check.py` 脚本生成前五列，后面三列数据经由教务系统探查后手工填入。表格里的17位同学有学籍异动记录，“降级”这种学籍异动形式会影响清洗后期[修读年限]列的计算，因此特别生成此临时表，统一修正数据库中降级同学的[修读年限]数据。

**项目背景说明：** 学校教务系统会默认给所有休学、降级学生的 '年级' 字段+1，因此清洗后分析宽表 `dim_student` 中的[grad_status]列（意图体现学生的修读年限以及最后是否毕业，由[毕业年份]列 - [年级]列得到）对于降级学生来说不再准确。（降级同学的修学时长往往是五年或六年）

下面通过举例来说明这个表格存在的必要性：<br>
1️⃣ 【常规学生】学号2013062301 的学生是2013年入学的、2013级，毕业于2017。按照 'grad_status' 列计算逻辑，2017-2013 = ‘四年毕业’；<br>
2️⃣ 【休学（后复学）学生】学号2013062301 的学生在大学期间休学一年后再复学，'年级' 字段被教务处调整为2014级，毕业于2018。按照 'grad_status' 列计算逻辑，2018-2014 = ‘四年毕业’，也是符合实际情况的，因为该同学中间一年休学，实际在校学习的时间就是四年；<br>
3️⃣ 【降级学生】学号2013062301 的学生在大四下学期发现学分修不满，无法毕业，只能降级再读一年，于是'年级' 字段被教务处调整为2014级，假设该同学在下一学年把剩下的学分修满，于2018年毕业。按照 'grad_status' 列计算逻辑，2018-2014 = ‘四年毕业’，这与实际情况不符，因为他实际在校学习了五年时间，所以这位同学应该属于’五年异常’。<br>

**处理逻辑：**
* 根据表格前部的信息手工从教务系统找到每位同学
* 写入[专业]和（学籍异动的）[真实情况] 字段
* 筛选同时符合 "xx设计" 和 "降级" 这两个条件的学生
* 新建一列，在符合筛选条件的同学所在的行写入 '是' 完成标记，给其他的学籍异动同学标记'否'

| 字段名称 | 数据类型 | 业务含义 | 缺失值/异常值状态声明 |
| :--- | :--- | :--- | :--- |
| 学号 | INTEGER | 学生唯一ID | 主键，无缺失 |
| 姓名 | TEXT | 学生姓名 | 无缺失 |
| 年级 | INTEGER | 教务系统导出的‘年级’字段 | 无缺失，常规情况下学生的“入学年份”=“年级”。但是如果学生休学、降级，则“年级”字段与“入学年份”字段不相关 |
| 入学年份 | TEXT | 取学号前四位数字得到（df_raw_grad_stage['入学年份'] = df_raw_grad_stage['学号'].str[:4]） | 无缺失 |
| 毕业年份 | FLOAT | 学生获得毕业证书的时间，教务系统导出数据 | 缺失2，存在2个学生未毕业 |
| 专业 | TEXT | 学生所在的专业 | 无缺失；美术学（师范）、环境设计、视觉传达设计、服装与服饰设计 |
| 真实情况 | TEXT | 学生的学号前四位数字与年级字段不匹配的原因，该数据由手工从教务系统逐一探查得到 | 无缺失；休学、降级、外国留学生 |
| 是否需要手动校准在校年限 | TEXT | 筛选出设计系的降级同学填入“是” | 无缺失；是，否 |


## 3. ETL总结
**处理逻辑：**
* 统一对四个表格进行了多维清洗（清理字符串空格、剔除非设计类的学生数据、逐一调查和处理缺失值、确认无重复记录、数据类型转换）
* 针对清洗过程中输出的缺失值及异常情况逐一进行穿透式探查、修复
* 在拼接完成三个分析宽表( `dim_student` . `fact_major_reassign` , `fact_gpa_records` ) 之后、导入进sql数据库之前对 [uid] 学号列进行哈希加密脱敏
* 清洗后的映射关系及具体情况如下：

## 4. 清洗后分析宽表 (Processed Analytical Table)
**表名：** `dim_student`

**说明：**  该表主要展现2014-2017级各个设计专业学生的基本情况，包括[学号(uid)]、[分流前班级(year_1_class_num)]、[分流后班级(grad_class_num)]、[入学年份(enroll_year)]和[毕业详情(grad_status)]字段的数据.（注：2014级学生不涉及大类分流，因此该年级学生[分流前班级]和[分流后班级]字段的内容是一致的）

**合并表格执行摘要:**
* 纵向拼接执行顺利 (纵向拼接`df_2014_1y_gpa_ready`的[学号]列、[班级]列 与 `df_stg_reassign_ready` 的[学号]列、[所在班级]列)
* 执行 'inner join' 模式的横向拼接时，发现共有17位学籍异常的学生被排除，经探查，这17位同学均不属于本次分析项目的观察对象，因此后续未进行修复。以下是被排除学生的具体情况：<br>
    * 有大一数据、无毕业数据：5人
        * 5位同学的数据存在在 `df_2014_1y_gpa_ready` 和`df_stg_reassign_ready`里，在 `df_raw_grad_ready`中却没有找到
        * 经返回教务系统探查，发现其中4位同学自大二学年起转专业至美术系、1位同学的数据没有在教务系统里找到。
        * 结论：这5位同学不属于分析对象。

    * 无大一数据、有毕业数据：12人
        * 12位同学的数据存在在 `df_raw_grad_ready`里，在 `df_2014_1y_gpa_ready` 和`df_stg_reassign_ready`中却没有找到
        * 经返回教务系统探查，发现其中7位同学自大三学年起由外校转入我校就读；1位外国同学没有参与专业分流志愿填报；1位同学是2013年入学的休学学生；3位同学是2014年入学的休学学生，在复学后加入2015级学习。
        * 结论：这12位同学不属于分析对象。（7位转学生与1位外国学生未经历大类分流志愿填报；1位2013年入学的学生不在2014-2017的分析范围内；3位2014级同学（未经历大类分流志愿填报，其学业表现应作为对照组，但从教务系统导出绩点和毕业数据时他们属于2015级）他们的存在会给"经历了大类分流的2015级学生"这个分析总体引入噪音。
* 删除[年级]列数据最后的“.0” (经由cast_types()从 FLOAT 类型转成字符串时保留了“.0”) 
* 确认已囊括所有需要列后，删去原表中不需要的数据列
* 修改列名
* 使用哈希将[uid]列数据脱敏

**表名：** `fact_gpa_records`

**说明：** 该表格主要展现了2014-2017级设计专业学生的平均学分绩点，每位同学有两条成绩记录，分属不同学段：第一学年课程的平均学分绩点和第二至四学年课程的平均学分绩点

**前置处理逻辑：**
* 纵向拼接从`df_2014_1y_gpa_ready`和`df_stg_reassign_ready`提取的[学号]和[平均学分绩点]列，得到`temp_y1_gpa`
* 提取`df_raw_last3y_gpa_ready`的[学号]和[平均学分绩点]列，得到`temp_y234_gpa`
* 给两个`temp`表各新增一列[academic_period]，分别填充'第一学年'和'第二至四学年'字段后将两个表格纵向拼接
* 修改列名
* 使用哈希将[uid]列数据脱敏
* 删除17位在构建 dim_student 表格时被inner join排除的学生的GPA数据
